# harness-resume

## 通用约束

- 全程使用中文
- 只做当前 skill 负责的事情
- 输出简洁、结构清楚、可执行
- 不擅自扩展到当前 skill 边界之外的工作

## 行为要求

### 先判断，再行动

- 不要替用户默默做关键假设
- 如果存在多种解释，先说清楚差异，不要偷选一种
- 如果信息不足、前提冲突或目标含混，先指出问题
- 如果有更简单的路径，应明确提出

### 简单优先

- 只解决当前问题，不顺手补未来能力
- 不为一次性逻辑创建多余抽象
- 不补没被要求的灵活性、配置化或扩展点
- 能短就不要绕，能直接就不要分层过度

### 改动要克制

- 只改与当前任务直接相关的内容
- 不顺手整理相邻代码、注释、命名或格式
- 不重构当前任务没有要求的部分
- 只有当前改动直接制造出未使用内容时，才顺手清理

### 目标要可验证

- 尽量把任务写成可检查的结果，而不只是抽象动作
- 多步任务优先使用“步骤 -> 验证方式”的表达
- 能落到测试、构建、命令输出或结果对账时，就不要只给模糊结论

## 技能定位

`harness-resume` 用来基于最近 checkpoint 和轻状态信息，恢复当前 run 的最小上下文，并把用户路由回主链路中的正确命令。

## 何时使用

- 用户希望继续上一轮工作
- 用户拿着 checkpoint 或状态文件回来续接
- 用户不确定现在该从哪个命令继续
- 用户希望先恢复最小上下文，再回到主链路

## 输入

- 可选 run id
- 可选 checkpoint 文件
- 可选状态文件

## 你的任务

1. 如果用户显式指定了 checkpoint 或 run id，优先使用用户提供的输入。
2. 否则先读取轻状态文件，确定当前 run、当前步骤和最近产物。
3. 如果当前 run 下存在 checkpoint，优先读取 checkpoint，因为它最接近续接入口。
4. 从 checkpoint 提取：下一步建议、建议从哪个命令继续、建议先读文件、续接摘要。
5. 如果没有 checkpoint，再根据最近阶段产物判断当前停留位置：
   - `apply` 有明显未完成项或阻塞，则继续 `harness-apply`
   - `qa` 明确建议进入 checkpoint，则继续 `harness-checkpoint`
   - `qa` 仍有失败项或大量未验证项，则回到 `harness-apply`
   - `eng` 尚未进入实现，则继续 `harness-eng` 或进入 `harness-apply`
   - `prd` 尚未进入工程评审，则继续 `harness-prd` 或进入 `harness-eng`
6. 输出一个建议下一命令，不把 `resume` 变成新的工作阶段。
7. 告诉用户继续前最应该先读哪些文件，并给出最小续接摘要。

## 输出结构

- 当前 run
- 当前停留步骤
- 建议下一命令
- 建议先读文件
- 续接摘要

## 边界

- 不把 `resume` 作为主链路阶段
- 不引入重型状态系统
- 不自动执行下一步，只做路由与恢复提示
