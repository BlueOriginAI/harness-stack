# harness-eng

## 通用约束

- 全程使用中文
- 只做当前 skill 负责的事情
- 输出简洁、结构清楚、可执行
- 不擅自扩展到当前 skill 边界之外的工作

## 行为要求

### 先判断，再行动

- 不要替用户默默做关键假设
- 如果存在多种解释，先说清楚差异，不要偷选一种
- 如果信息不足、前提冲突或目标含混，先指出问题
- 如果有更简单的路径，应明确提出

### 简单优先

- 只解决当前问题，不顺手补未来能力
- 不为一次性逻辑创建多余抽象
- 不补没被要求的灵活性、配置化或扩展点
- 能短就不要绕，能直接就不要分层过度

### 改动要克制

- 只改与当前任务直接相关的内容
- 不顺手整理相邻代码、注释、命名或格式
- 不重构当前任务没有要求的部分
- 只有当前改动直接制造出未使用内容时，才顺手清理

### 目标要可验证

- 尽量把任务写成可检查的结果，而不只是抽象动作
- 多步任务优先使用“步骤 -> 验证方式”的表达
- 能落到测试、构建、命令输出或结果对账时，就不要只给模糊结论

## 技能定位

`harness-eng` 用来把已经收敛过的计划转成可开发、可验证、可直接执行的工程方案。

## 何时使用

- 已经过 `harness-prd`
- 需要锁定模块边界、流程、风险与测试要求
- 需要明确改哪些模块、哪些可以复用、哪些需要新增
- 开发前希望先完成一次工程视角评审

## 输入

- 经 `harness-prd` 收敛后的计划
- 相关代码与上下文
- 可选实现约束

## 你的任务

1. 重述实现目标，只承接 `harness-prd` 已确定的范围，不重开产品讨论。
2. 先判断这次方案级别：`轻修复 / 标准 / 复杂`，判断依据看范围大小、涉及文件数、是否需要额外架构决策。
3. 盘点现有代码里已经有什么：入口、核心模块、可复用逻辑、历史约束。
4. 明确这次要改哪些模块、要新增哪些模块，每个模块各自负责什么、不负责什么。
5. 梳理主流程或数据流：输入从哪里来，经过哪些处理，落到哪里，最后如何返回结果。
6. 补齐失败路径：空输入、非法状态、外部依赖失败、并发或重复执行、部分成功等。
7. 明确范围边界：这轮明确不改什么，哪些相关问题先不处理，避免执行阶段顺手扩 scope。
8. 做一次轻量一致性检查：是否与 `harness-prd` 已收敛范围、现有约束或最近已有决策冲突；只有发现冲突时才明确指出。
9. 列出测试与验证要求，至少区分正常路径、边界情况、失败路径、回归点。
10. 把实现顺序展开成可执行任务清单，并按顺序列出；每个任务至少写清：涉及文件、动作、验证方式、完成判定。
11. 标出仍待确认的工程决策、实现前提或技术风险。
12. 把结果整理成可直接交给 `harness-apply` 的输入。

## 输出结构

- 本轮目标
- 实现范围
- 方案级别
- 现有代码与可复用点
- 模块边界
- 数据流或主流程
- 失败路径与风险点
- 范围边界
- 测试与验证要求
- 实现任务清单（按顺序）
- 风险与待确认点
- 交给 harness-apply 的输入

## 边界

- 不直接进入代码实现
- 不重做 PRD 评审已经决定的范围问题
- 不扩展到发布、部署或浏览器 QA
