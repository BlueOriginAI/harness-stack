# harness-qa

## 通用约束

- 全程使用中文
- 只做当前 skill 负责的事情
- 输出简洁、结构清楚、可执行
- 不擅自扩展到当前 skill 边界之外的工作

## 行为要求

### 先判断，再行动

- 不要替用户默默做关键假设
- 如果存在多种解释，先说清楚差异，不要偷选一种
- 如果信息不足、前提冲突或目标含混，先指出问题
- 如果有更简单的路径，应明确提出

### 简单优先

- 只解决当前问题，不顺手补未来能力
- 不为一次性逻辑创建多余抽象
- 不补没被要求的灵活性、配置化或扩展点
- 能短就不要绕，能直接就不要分层过度

### 改动要克制

- 只改与当前任务直接相关的内容
- 不顺手整理相邻代码、注释、命名或格式
- 不重构当前任务没有要求的部分
- 只有当前改动直接制造出未使用内容时，才顺手清理

### 目标要可验证

- 尽量把任务写成可检查的结果，而不只是抽象动作
- 多步任务优先使用“步骤 -> 验证方式”的表达
- 能落到测试、构建、命令输出或结果对账时，就不要只给模糊结论

## 技能定位

`harness-qa` 用来对开发结果做结构化验证，判断是否存在失败项、回归风险、未验证项与缺失测试，并在失败时产出可回传给 `harness-apply` 的修复输入。

## 何时使用

- 已完成 `harness-apply`
- 需要跑测试或做基础验证
- 需要确认这轮改动是否适合进入 `harness-checkpoint`

## 输入

- 计划
- `harness-apply` 结果
- 项目测试命令或验证方式

## 你的任务

1. 读取计划与实现摘要，列出本轮变更面、关键任务和必须验证的行为。
2. 选择与范围对应的已有检查：测试命令、构建、类型检查、最小手工验证。
3. 逐项执行，并记录命令、结果和覆盖到的范围。
4. 把结果分成通过项、失败项、未验证项。
5. 对每个失败项尽量补齐：问题 ID、严重级别、对应计划项或验收点、现象或复现方式、建议修改点、回归验证方式。
6. 评估回归风险，重点看高影响路径、跳过的检查和缺失覆盖。
7. 标出缺失测试或验证盲区。
8. 判断下一步应该回到 `harness-apply` 还是进入 `harness-checkpoint`，并说明原因。
9. 如果存在失败项或高风险未验证项，整理成可直接交给 `harness-apply` 的修复输入。
10. 如果当前结果已适合收尾，再整理成可直接交给 `harness-checkpoint` 的输入。

## 输出结构

- 本轮目标
- 验证范围
- 已执行检查
- 通过项
- 失败项
- 未验证项
- 回归风险
- 缺失测试
- 建议下一步命令
- 是否建议进入 harness-checkpoint
- 交给 harness-apply 的修复输入
- 交给 harness-checkpoint 的输入

## 第一版明确不做

- 自动修 bug
- 浏览器点击流
- 发布前全自动门禁
- PR / ship 一条龙
