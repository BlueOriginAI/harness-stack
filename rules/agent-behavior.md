# Agent Behavior Rules

## 目的

本文件定义 harness-stack 所有 skill 默认继承的通用行为约束。

它不负责说明某个命令的流程，而是约束 agent 在任何流程里都不要犯低级错误。

## 总原则

- 先判断，再行动
- 简单优先，不预支复杂度
- 改动要克制，只碰当前任务必须碰的部分
- 每一步都尽量给出可验证结果，而不是停留在模糊完成感

## 1. 先判断，再行动

- 不要替用户默默做关键假设
- 如果存在多种解释，先说清楚差异，不要偷选一种
- 如果发现信息不足、前提冲突或目标含混，先停下来指出问题
- 如果有更简单、更直接的路径，应明确提出，不盲从复杂方案

## 2. 简单优先

- 只解决当前问题，不顺手增加未来也许会用到的能力
- 不为一次性逻辑创建多余抽象
- 不补用户没要求的灵活性、配置化或扩展点
- 不为内部不可能发生的情况堆过量防御代码
- 如果一个方案明显可以更短、更直白，就不要保留更绕的写法

## 3. 改动要克制

- 只改与当前任务直接相关的代码、文案或结构
- 不顺手整理相邻代码、注释、命名或格式
- 不重构没有被当前任务要求的部分
- 如果你发现无关的坏味道、死代码或历史问题，可以提示，但不要擅自清理
- 只有当当前改动直接制造出未使用 import、变量或函数时，才顺手清掉它们

## 4. 目标要可验证

- 不要只说“做完了”，要尽量说明如何验证已经做对
- 对多步任务，优先写成“步骤 -> 验证方式”的形式
- 能落到测试、构建、类型检查、命令输出或明确结果对账时，就不要只给抽象描述
- 如果任务本身缺少完成标准，应先补出最小可验证标准

## 何时可以放松

对非常小、非常直接的任务，可以减少形式化表达，不必每次都展开完整步骤。

但即便是小任务，也不要：

- 乱猜需求
- 过度设计
- 顺手改无关内容
- 在没有验证的情况下宣称完成
