# 术语表

## harness-prd

中文定位：问题定义 / 用户价值 / 范围收敛。

不是写 PRD 文档工具，也不是自由 brainstorm。

它先把真实痛点、用户对象、现状替代、最小切口和当前版本边界收清楚。

## harness-eng

中文定位：把 PRD 输出转成可执行开发输入。

负责先盘点现有代码，再明确这次改哪里、主流程怎么走、失败路径在哪里、该怎么验证、这轮明确不改什么，并整理成可直接交给 `harness-apply` 的实现任务清单与输入。

## harness-apply

中文定位：把工程方案转成验证输入。

不是自由发挥写代码，而是基于已评审计划推进实现，按任务清单落地，并把已完成项、偏差、未完成项和待验证点留下来。

## harness-qa

中文定位：把实现结果转成 checkpoint 输入，或在失败时转成回炉修复输入。

不是重型浏览器 QA 平台，也不是自动修复机器人。第一版只负责按范围记录通过项、失败项、未验证项、风险与缺失测试，给出下一步应该回到 `harness-apply` 还是进入 checkpoint 的判断，并在需要时整理修复输入。

## harness-checkpoint

中文定位：对账结果与续接快照。

重点是把计划、实现和 QA 三段结果对齐，留下可恢复的阶段节点、下一步建议和推荐续接入口，不把过程管理做成重状态系统。

## harness-resume

中文定位：跨会话续接入口。

它优先读取最近 checkpoint，必要时再结合轻状态信息，判断当前 run 停在哪一步、下一步该从哪个命令继续。

## 粗计划

用户手写的初始计划或需求草稿，是第一版工作流的起点。

## 薄命令

命令文件只负责入口、用途、路由与引用，不塞入大量细节。

## 厚流程

workflow 文件承载主要执行逻辑，是系统行为的主体。
