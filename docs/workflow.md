# 工作流

## 主链路

harness-stack 第一版主链路：

`用户先写粗计划并交给 harness-prd -> harness-eng -> harness-apply -> harness-qa -> harness-checkpoint`

如果 `harness-qa` 出现失败项或高风险未验证项，会把修复输入回传给 `harness-apply`；修复后再回到 `harness-qa`。

## 每一步做什么

### 1. 用户先写粗计划并交给 harness-prd

用户先提供一个粗计划、需求草稿或设计草稿，然后直接进入 `harness-prd`。

第一版不负责从 0 到 1 自动起草计划。

### 2. harness-prd

从问题定义、用户价值与范围角度审视草稿：

- 问题是不是定义对了
- 谁最痛、场景够不够具体
- 用户现在怎么替代解决
- 当前版本的最小切口是什么
- 哪些内容应该先做
- 哪些内容应该明确不做

### 3. harness-eng

从工程、架构、测试角度把计划收敛成可开发方案，并把 PRD 输出转成开发输入：

- 接住 `harness-prd` 的本轮目标和本轮范围
- 判断这次属于轻修复、标准还是复杂方案
- 盘点现有代码与可复用点
- 明确模块边界、主流程和失败路径
- 定义范围边界与测试要求
- 产出按顺序展开的实现任务清单
- 输出交给 `harness-apply` 的输入

### 4. harness-apply

按已收敛的计划执行开发，并把工程方案转成验证输入：

- 接住 `harness-eng` 的实现范围、范围边界和实现任务清单
- 按主路径和高风险点推进实现
- 记录已完成项、偏差、未完成项和风险
- 输出交给 `harness-qa` 的输入

### 5. harness-qa

开发后做验证，并把验证结果转成 checkpoint 输入或修复输入：

- 接住 `harness-apply` 的已完成项和待验证点
- 列出本轮必须验证的范围
- 区分通过项、失败项、未验证项
- 判断回归风险与缺失测试
- 如果存在失败项或高风险未验证项，整理成可回传给 `harness-apply` 的修复输入
- 如果验证结果已收敛，再输出交给 `harness-checkpoint` 的输入

### 6. harness-checkpoint

保存本轮工作检查点：

- 接住计划、实现和 QA 三段结果
- 对账计划和实际完成
- 说明与计划的差异
- 归纳未完成项、遗留问题和风险状态
- 明确下一步建议与续接入口

关于当前交接方式与是否采用文件化状态保存，见 `docs/state-model.md`。

## 最小示例

下面用同一个例子，演示主链路如何串起来。

### 示例题目

给设置页增加“退出登录”按钮，点击后清空本地会话并跳回登录页。

### 1. 交给 harness-prd 的粗计划

用户可以直接给出这样一段：

> 想在设置页加一个退出登录按钮。现在用户想退出只能清浏览器存储，不够直观。我希望这次先做最小版本：能退出当前会话并回到登录页，不做“退出所有设备”。

### 2. harness-prd 之后

`harness-prd` 会把这段草稿收敛成：

- 本轮目标：提供最小退出登录能力
- 本轮范围：设置页按钮 + 清空当前会话 + 回到登录页
- 明确不做：退出所有设备、复杂账户管理扩展
- 交给 `harness-eng` 的输入：一份较稳定的本轮目标与范围

### 3. harness-eng 之后

`harness-eng` 会继续把它变成可执行开发输入：

- 判断方案级别：标准
- 写清模块边界、主流程、失败路径、范围边界
- 产出实现任务清单，例如：
  1. 在设置页增加退出登录按钮
  2. 补齐最小登出逻辑，清空会话状态
  3. 登出后跳转登录页并验证回归路径
- 把这些内容交给 `harness-apply`

### 4. harness-apply 之后

`harness-apply` 会记录这轮实际实现结果，例如：

- 已完成：按钮已加上，点击后会跳转登录页
- 偏差：原本以为已有统一 `logout()` 方法，实际补了一个最小公共方法
- 风险与待验证点：退出后刷新页面是否还残留旧用户名
- 交给 `harness-qa` 的输入：应重点验证按钮展示、会话清理、跳转与受保护页面回归

### 5. harness-qa 之后

如果 QA 首轮发现问题，结果可能是：

- 通过项：按钮展示正常、点击后成功跳转登录页
- 失败项：退出后刷新页面，顶部用户名偶发仍显示上一个用户
- 建议下一步命令：`harness-apply`
- 交给 `harness-apply` 的修复输入：补清理内存 user store，并重新验证退出后刷新场景

修完后再跑一次 `harness-qa`，确认失败项消失。

### 6. harness-checkpoint 之后

当 QA 已收敛，再进入 `harness-checkpoint`：

- 对账计划与实际完成
- 记录这轮额外补了最小公共登出方法
- 留下未完成项，例如自动化测试和确认弹窗
- 给出下一步建议，例如“如果继续增强，先补自动化测试”
- 生成可供跨会话续接的摘要

## 第一版边界

第一版先不覆盖：

- 自动起草计划
- 重型 debug / investigate
- 独立 review
- ship / deploy
- 浏览器自动 QA
