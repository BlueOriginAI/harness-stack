## 示例（参考写法）

以下示例只演示写法，不限制真实内容。

### 示例场景

验证“设置页增加退出登录按钮”这轮改动。

## 本轮目标

确认用户可以从设置页退出登录，并且不会残留登录态。

## 验证范围

- 设置页退出按钮展示
- 点击后本地会话清理
- 跳转登录页
- 返回受保护页面的回归行为

## 已执行检查

- `pnpm test --filter session`：通过
- `pnpm lint`：通过
- 手工验证：登录 -> 设置 -> 退出 -> 返回受保护页

## 通过项

- 设置页已出现退出登录按钮
- 点击后可跳转到 `/login`
- 返回受保护页面时会重新要求登录

## 失败项

1. `QA-001`
   - 严重级别：Major
   - 对应计划项或验收点：任务 2 / “登出后不存在可继续复用的会话状态”
   - 现象或复现方式：退出后刷新页面，顶部用户名偶发仍显示上一个用户
   - 建议修改点：检查内存中的 user store 是否同步清空，确认不是只清 localStorage
   - 回归验证方式：退出后刷新 `/login` 和任一受保护页，确认不再出现旧用户信息

## 未验证项

- 移动端窄屏下按钮位置未检查

## 回归风险

- 登录态缓存与 UI store 可能存在双写，若只清一个层次会出现残留

## 缺失测试

- 缺少“退出后刷新页面不残留用户信息”的自动化测试

## 建议下一步命令

- `harness-apply`

## 是否建议进入 harness-checkpoint

- 否
- 原因：仍有 Major 级失败项，需先修复后再验证

## 交给 harness-apply 的修复输入

- 先修 `QA-001`
- 重点检查会话清理是否同时覆盖 localStorage、memory store、页面初始化逻辑
- 修复后最少回归：
  - 登录 -> 设置 -> 退出 -> 刷新登录页
  - 直接访问受保护页，确认被拦回登录页

## 交给 harness-checkpoint 的输入

- 暂不进入 checkpoint
- 待 `QA-001` 修复并复验通过后再总结
